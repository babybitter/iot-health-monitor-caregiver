# IoT 健康监测护工端

面向病区或照护机构护工的微信小程序。项目复用患者端已经验证的原生小程序、HTTP、MQTT、阈值判断、本地缓存和 ECharts 能力，并按护工的值班场景重新组织为工作台、智能监控、任务和我的四个一级入口。

当前仓库默认使用结构化模拟数据，便于在没有护工业务接口时完整演示任务、考勤、告警确认和照护记录流程。患者生命体征的真实 HTTP 接口及 MQTT 实时通道均保留适配入口，不包含密钥或真实登录凭据。

## 技术栈

- 微信原生小程序：JavaScript、WXML、WXSS、Glass EASEL
- 微信开发者工具，基础库 `3.8.10`
- ECharts 微信小程序画布组件，用于单指标历史趋势
- `App.globalData` 与轻量事件 Store，用于跨页选中患者和网络状态
- `wxStorage`，用于模拟模式下的任务、考勤、告警和照护记录持久化
- 原生 `wx.request` 与 `wx.connectSocket`，不引入大型运行时依赖
- npm 仅用于执行仓库自检脚本

│  ├─ patients/         负责患者筛选与分页
│  ├─ records/          照护记录列表
│  ├─ record-edit/      照护记录表单与校验

## 环境要求

- Node.js 18 或更高版本
- npm 9 或更高版本
- 微信开发者工具，建议基础库不低于 `3.8.10`
- 接入真实接口时，需要在微信公众平台配置合法 request 与 socket 域名

## 安装与本地运行

```bash
npm install
npm run build
```

然后在微信开发者工具中导入当前 `caregiver` 目录。仓库使用 `touristappid`，可以用于本地预览；正式开发和上传前应在 `project.config.json` 中替换为护工端小程序 AppID。

## 检查与构建

```bash
npm run lint
npm test
npm run build
```

- `lint` 校验 JSON、页面注册、组件引用、JavaScript 语法、AI 路由残留、WXML 不兼容表达式、Emoji 和明显超宽固定布局。
- `test` 覆盖指标阈值、患者分页、趋势数据、任务状态、签到签退和照护记录校验。
- `build` 串行执行上述两类检查。原生小程序最终编译仍由微信开发者工具完成。

## 配置说明

默认配置位于 `config/index.js`，其中 `dataMode` 为 `mock`。微信开发者工具会在打包期解析静态 `require`，因此项目不会引用一个可能缺失的本地配置模块。

接入真实环境时，由发布流程根据 `config/local.example.js` 的结构写入非敏感地址与模式配置。MQTT 用户名、密码和业务 Token 不应编译进小程序，应在用户登录后由受信任的后端接口下发并仅保存在运行时内存中。

项目没有使用 `.env`，因为微信小程序运行时不会自动读取 Node.js 环境变量。仓库中的配置示例不包含可用密钥。

## 已实现功能

- 工作台：班次状态、签到签退、异常摘要、负责患者、今日任务和快捷监控
- 智能监控：患者切换、生命体征阈值、输液余量与滴速、异常确认、历史趋势、设备在线状态和受控蜂鸣提醒
- 任务：状态筛选、分页、详情、开始任务、完成确认和照护记录入口
- 负责患者：隐私化名称、状态筛选、搜索、分页、监控与记录入口
- 照护记录：患者筛选、分页、新增表单、长度限制、必填校验和防重复提交
- 考勤：当前班次、签到签退、二次确认和历史记录；当前版本不伪造定位结果
- 我的：资质证书临期提醒、培训安排与历史结果
- 通用状态：加载、空数据、请求失败、网络离线、刷新和操作反馈

## API 与模拟数据组织

`services/repository.js` 是页面唯一使用的数据入口：

- 护工专属业务暂由 `mocks/data.js` 与本地缓存实现，数据结构与页面解耦。
- `remote` 模式下，生命体征和历史趋势调用患者端现有的 `/api/latest/:deviceId` 与 `/api/history/:deviceId`。
- 后续护工服务提供患者分配、任务、考勤、记录、证书和培训接口后，在 repository 对应方法中替换实现即可，页面不需要改写。
- `services/request.js` 统一处理超时、HTTP 状态、业务失败和可选 Bearer Token。
- `services/realtime.js` 使用 SocketTask 接入 MQTT 3.1.1，避免患者端全局 Socket 监听器重复注册的问题。

## 主题与布局

主题变量统一位于 `styles/theme.wxss`。页面使用弹性布局、Grid 的 `minmax(0, 1fr)`、文本截断与自动换行，所有主要容器均设置 `min-width: 0` 和边界裁剪。自定义导航读取状态栏与胶囊位置，底部一级页面为原生 tabBar，内容区为安全区域预留空间。

## 与患者端的主要差异

- 首页由按传感器分类改为值班风险工作台，首屏优先异常、患者、任务和班次。
- 监控从查看本人改为一次查看一个服务对象，可切换患者、指标和时间范围。
- 保留生命体征、输液、趋势、设备与异常判断，新增告警确认和照护记录衔接。
- 移除 AI 助手的页面、路由、依赖、图标和配置。
- 患者资料、医生和家属入口不作为护工端一级能力，避免不必要的敏感信息暴露。
- 绿色健康主题调整为克制的蓝色值守主题，并为成功、关注、紧急和离线建立统一状态色。

## 当前限制

- 护工专属后端接口尚未提供，因此任务、考勤、告警处理和照护记录默认保存在本机缓存，不能跨设备同步。
- 定位考勤未启用，界面明确显示当前不采集定位；接入真实考勤接口后再按机构规则申请权限。
- 设备蜂鸣提醒只有在本地配置 MQTT 且连接成功后才会发送；模拟模式不会伪造发送成功。
- 证书上传和在线考试未实现，页面只展示已有数据和安排。
- 正式上传前需要配置护工端 AppID、服务器合法域名、登录换 Token 和后端权限控制。
